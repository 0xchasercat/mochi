The short version: minor bumps are additive, major bumps are real changes, profile IDs are split into stable and placeholder buckets. The long version is below.

See also: [Changelog](/docs/reference/changelog), [Invariants](/docs/reference/invariants), [Limits](/docs/reference/limits).

## The additive-minor-bump policy

`@mochi.js/core` minor bumps (and every other `@mochi.js/*` package) are **additive**. We never remove a public export without a major bump. The contract:

- **Patch (`0.1.X`)** — bug fixes, performance, no API change. Always safe to upgrade.
- **Minor (`0.X.0`)** — new public exports, new public methods on existing classes, new optional fields on existing types. Existing call sites keep compiling.
- **Major (`X.0.0`)** — public-export removal, parameter-shape changes, default-value changes, or breaking changes to invariants (PLAN.md §2). Documented per-package in `packages/<name>/CHANGELOG.md`.

Independent versioning is per [decision #14](https://github.com/0xchasercat/mochi/blob/main/PLAN.md). Each `@mochi.js/*` package on npm carries its own semver; Changesets enforces. The "release" of mochi as a project is the set of `@mochi.js/*` packages published together at a tag.

## How to read `CHANGELOG.md`

The aggregated [`CHANGELOG.md`](https://github.com/0xchasercat/mochi/blob/main/CHANGELOG.md) at the repo root summarizes each release across the workspace. Per-package detail lives in `packages/<name>/CHANGELOG.md` (auto-generated by [Changesets](https://github.com/changesets/changesets)).

The site mirror is at [Changelog](/docs/reference/changelog).

For each release we document:
- Affected packages and their version bumps.
- New public surfaces (with a link to the doc page).
- Known limits introduced by the release (mirrored to [Limits](/docs/reference/limits)).
- Install-blocking issues (e.g. v0.1.1 hot-fix for `workspace:*`).

## Major-bump checklist (what to expect at v1.0)

v1.0 is the production-ready inflection point. Targeted changes:

- **Stability commitment.** v1 surfaces commit to the stricter no-removal-without-RFC contract. Today that's `mochi.launch`, `Session`, `Page`, `Page.humanClick / humanType / humanScroll`, `Session.cookies`, `session.fetch`, `Page.localStorage / sessionStorage`, `Page.grantAllPermissions`, `Page.screenshot`. Anything not on that list at v1.0 freeze is an explicit experimental.
- **Renames (if any) are documented.** Today, no public surface is slated for rename at v1. Internal-only items can churn.
- **Behavioral recorder** — `humanClick(sel, { trace })` accepts a recorded trace alongside the synthesizer. The forward-compatible parameter is already in the contract; the recorder lands in v1.x.
- **Profile catalog growth.** Additional real-device captures from the harvester corpus. Placeholder IDs (see below) may flip to real captures.
- **Probe Manifest schema.** Defined at `schemas/probe-manifest.schema.json`. v1.0 freezes the schema version; future bumps follow semver and are tracked in `CHANGELOG.md`.

Anything that would violate an invariant (PLAN.md §2 / I-1 through I-8) is by definition not in the v1.0 plan. Migrations across major-bumps will get a dedicated section here when they exist.

## Upgrade from v0.7 → v0.8 (mochi.connect, profile: null, final wreqPreset removal)

`@mochi.js/core@0.8.x` lands four user-visible changes. None break existing call sites; everything is additive at the API surface.

**1. New: `mochi.connect(opts: ConnectOptions): Promise<Session>`.** Attach to a CDP browser mochi did NOT spawn — BrowserBase, Browserless, dockerised Chromium, re-attach, your own patched Chrome. Mirrors `puppeteer.connect`'s shape: takes a `wsEndpoint` (direct WebSocket URL) or a `browserURL` (HTTP base, mochi GETs `/json/version` to discover the WS URL). `session.close()` disconnects the WebSocket without killing the browser. See [Connect to an existing Chrome](/docs/guides/connect-existing-chrome) and [`api/core` → `function connect`](/docs/api/core#function-connectopts-connectoptions-promisesession).

**2. New: `profile: null` no-spoof mode.** Both `mochi.launch` and `mochi.connect` now accept `profile: null` to skip the matrix derivation, the inject pipeline, and every CDP override (`Network.setUserAgentOverride`, `Emulation.setTimezoneOverride`, `Emulation.setLocaleOverride`, every `addScriptToEvaluateOnNewDocument`). `Session.profile` surfaces as `null`. Behavioral synth falls back to `DEFAULT_BEHAVIOR` from `@mochi.js/behavioral`. The right shape when you've layered your own stealth elsewhere (a patched binary, your own inject pipeline, a remote browser that already spoofs) or when you want to drive a bare browser through mochi's `Page` / `Session` API.

**3. New: `DEFAULT_BEHAVIOR` constant.** Distinct from the longstanding `DEFAULT_BEHAVIOR_PROFILE`. `DEFAULT_BEHAVIOR` is the no-spoof-mode default (`wpm: 60`); `DEFAULT_BEHAVIOR_PROFILE` is the in-band default for matrix-derived sessions (`wpm: 65`). Both export from `@mochi.js/behavioral`.

**4. `wreqPreset` final cleanup.** The runtime ignored the field starting in 0.7; in 0.8 the documentation is firmly on "deprecated, kept in the schema for back-compat only." The `@mochi.js/net` and `@mochi.js/net-rs` packages remain unpublished.

**5. `ALL_BROWSER_PERMISSIONS` retuned for Chromium 148** (the retune landed in 0.7; 0.8.1 confirmed the CfT pin to `148.0.7778.97`). 39 entries.

**6. New: `connectWebSocketCdp` + `WebSocketCdpAdapter` building blocks** for tools that want to drive the WebSocket transport directly without the full `Session` lifecycle.

**7. New: `EXPLICIT_PROFILE_IDS`** const — the six captured-baseline profile IDs, exported publicly so downstream tooling can render the same list verbatim.

## Upgrade from v0.6 → v0.7 (Session.fetch routes through Chromium)

`@mochi.js/core@0.7.0` drops the Rust `wreq` HTTP layer. `Session.fetch` now routes through Chromium itself via CDP — JA4/JA3/H2 are real Chrome by definition because Chromium is the client. The two npm packages that wrapped wreq (`@mochi.js/net`, `@mochi.js/net-rs`) are deprecated and no longer published.

**Breaking change you have to know about — cookie inheritance.** `Session.fetch` now shares the session's cookie jar with the browser. A cookie set via `Page.goto` or `session.cookies.set` is sent on the next `Session.fetch` to the same origin automatically. Pre-0.7 the wreq path was cookieless. If your code relied on the cookieless behavior, the migration is one of:

- Set `init.credentials = "omit"` for the page-evaluate path (POST / non-GET).
- Clear the relevant cookies before the call (`session.cookies.set([])` or pattern-filtered).
- Pass the cookie explicitly via `init.headers["Cookie"]` only if you also clear the jar; the page's `fetch` will merge with what's in the jar otherwise.

**CORS applies for non-GET cross-origin calls.** Mechanism A (simple GET) bypasses CORS at the network layer; Mechanism B (POST / custom headers / body) routes through `page.evaluate("fetch")` and obeys the same CORS rules a user's browser would. Cross-origin POSTs to an endpoint without `Access-Control-Allow-Origin` will fail in 0.7 where they may have succeeded in 0.6.

**Body shapes.** `Blob`, `FormData`, and `ReadableStream` request bodies throw with a clear diagnostic. `string`, `ArrayBuffer` / typed arrays, and `URLSearchParams` continue to work. Richer bodies land in a follow-up PR.

**No cdylib install.** `bunx mochi pm trust @mochi.js/net-rs` and the `cargo build` fallback are gone. There is nothing to install or trust beyond Chromium itself.

**Profile fields.** `ProfileV1.wreqPreset` and `MatrixV1.wreqPreset` are deprecated; the runtime no longer reads them. The schema retains them for one release for migration and drops in 0.8.

**Permission list retuned for Chromium 148.** `ALL_BROWSER_PERMISSIONS` matches the `Browser.PermissionType` enum on Chromium 148. Removed: `accessibilityEvents`, `captureHandle`, `flash`, `videoCapturePanTiltZoom`. Added: XR cluster, `automaticFullscreen`, `cameraPanTiltZoom`, `capturedSurfaceControl`, `keyboardLock`, `pointerLock`, network-permission cluster, `smartCard`, `webPrinting`. Calls to `page.grantAllPermissions()` against an older Chromium will fall through with no behavior change; calls against 148 stop tripping the `Unknown permission type: accessibilityEvents` runtime error.

## Upgrade-from-v0.1 notes

The v0.2 wave-4 surfaces are **additive**. If you're on `@mochi.js/core` `0.1.0` or `0.1.1` and you upgrade to `0.1.4`, no existing code breaks; you just gain new APIs. Specifically:

- **Audio + canvas precomputed blobs (R-047, R-048)** — invisible to your code; the matrix produces them and the inject consumes them. Probes that previously slipped through the audio / canvas surface now match byte-exactly.
- **Cookie persistence** (`Session.cookies.{save,load}`) — additive.
- **DOM storage** (`page.localStorage.{get,set}`, `page.sessionStorage.{get,set}`) — additive; previously you'd `page.evaluate(() => localStorage.foo)`.
- **`page.grantAllPermissions(opts?)`** — additive.
- **`page.screenshot(opts?)`** — was `NotImplementedError` in earlier 0.1.x; now returns `Uint8Array` (default) or `string` (`encoding: "base64"`). If you had `try { ... } catch (NotImplementedError)` blocks around screenshots, the catch block is now dead code.
- **Init-script dual mechanism** — invisible to your code; closes the `addScriptToEvaluateOnNewDocument` source-attribution leak via `Fetch.fulfillRequest` body splice on Document responses.
- **Turnstile auto-click** — opt-in via `challenges: { turnstile: { autoClick: true } }`. Default `false`. No-op if you don't pass it.

See the [Changelog](/docs/reference/changelog) for the per-version detail.

## Workspace-protocol gotcha (pre-v0.1.1)

If you installed `@mochi.js/core@0.1.0` and saw `Workspace dependency not found` errors, that's the v0.1.0 publish bug: `workspace:*` references made it into the published `package.json`. Changesets / `npm publish` does not rewrite `workspace:*` to concrete semver — that is a pnpm/yarn courtesy npm doesn't replicate.

The fix is in v0.1.1+ via `scripts/rewrite-workspace-deps.ts` (publish-time pre-hook). **Upgrade to ≥ v0.1.1**:

```sh
bun add @mochi.js/core@latest @mochi.js/cli@latest
```

Affected packages bumped to `0.1.1`: `@mochi.js/core`, `@mochi.js/cli`, `@mochi.js/inject`, `@mochi.js/challenges`, `@mochi.js/behavioral`, `@mochi.js/harness`, `@mochi.js/profiles`. `@mochi.js/consistency` was a leaf package with no internal deps and remained at `0.1.0`.

## Profile-ID stability

The profile catalog is **split into two buckets** with different stability guarantees.

**Real-device baselines (stable IDs — depend on these):**
- `mac-m4-chrome-stable`
- `mac-chrome-stable`
- `mac-chrome-beta`
- `windows-chrome-stable`
- `linux-chrome-stable`
- `mac-brave-stable`

These are captured by `mochi capture` on real hardware, filtered by FingerprintJS Pro `suspectScore <= 20`, validated by harness Zero-Diff round-trip. The IDs are stable. Future patch / minor releases will not rename them; new captures of the same device class extend (e.g. a Mac M5 lands at `mac-m5-chrome-stable`, not by replacing `mac-m4-chrome-stable`).

**Placeholders (do NOT depend on these IDs — they may flip to real captures or be renamed):**
- `mac-m2-chrome-stable`
- `mac-m1-chrome-stable`
- `mac-intel-chrome-stable`
- `win11-chrome-stable`
- `win11-edge-stable`

These resolve to a generic synthesis that's consistency-clean but doesn't match any specific captured device. When a real Mac M2 / Mac M1 / Mac Intel / Win11 / Edge capture lands in the harvester corpus, the placeholder ID may either (a) become a real capture under the same ID, or (b) be renamed (e.g. `mac-m2-air-chrome-stable` if the captured machine is specifically a Mac M2 Air, leaving `mac-m2-chrome-stable` as the new placeholder for the device class).

If you want stability today, **use the six real-device IDs**. Track new captures via the [Limits](/docs/reference/limits) page.

## Schema versioning

The `MatrixV1` and `ProbeManifestV1` schemas carry their version in the type name. A future `MatrixV2` would land alongside `MatrixV1` (not as a replacement); old captures continue to deserialize cleanly. Same shape for `ProfileV1` → `ProfileV2`.

`@mochi.js/profiles` ships per-profile JSON validated against `schemas/profile.schema.json` and `schemas/probe-manifest.schema.json`. Schema diffs are documented in PLAN.md §6.

## What to do when a release breaks Zero-Diff

If a CI run on your fork breaks Zero-Diff after upgrading mochi, the canonical move is:

1. Check `packages/profiles/data/<id>/expected-divergences.json` for the affected profile — the diff may be a new GUID-class field that needs allowlisting.
2. Re-capture the baseline if you own the device — `mochi capture --output packages/profiles/data/<id>/`.
3. Open an issue tagged `harness`. The harness is the gate per invariant I-7; a Zero-Diff regression is treated as a real bug.

See [Probe Manifest](/docs/concepts/probe-manifest) for the harness-gate flow.

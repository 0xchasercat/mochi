The short version: minor bumps are additive, major bumps are real changes, profile IDs are split into stable and placeholder buckets. The long version is below.

See also: [Changelog](/docs/reference/changelog), [Invariants](/docs/reference/invariants), [Limits](/docs/reference/limits).

## The additive-minor-bump policy

`@mochi.js/core` minor bumps (and every other `@mochi.js/*` package) are **additive**. We never remove a public export without a major bump. The contract:

- **Patch (`0.1.X`)** â€” bug fixes, performance, no API change. Always safe to upgrade.
- **Minor (`0.X.0`)** â€” new public exports, new public methods on existing classes, new optional fields on existing types. Existing call sites keep compiling.
- **Major (`X.0.0`)** â€” public-export removal, parameter-shape changes, default-value changes, or breaking changes to invariants (PLAN.md Â§2). Documented per-package in `packages/<name>/CHANGELOG.md`.

Independent versioning is per [decision #14](https://github.com/0xchasercat/mochi/blob/main/PLAN.md). Each `@mochi.js/*` package on npm carries its own semver; Changesets enforces. The "release" of mochi as a project is the set of `@mochi.js/*` packages published together at a tag.

## How to read `CHANGELOG.md`

The aggregated [`CHANGELOG.md`](https://github.com/0xchasercat/mochi/blob/main/CHANGELOG.md) at the repo root summarizes each release across the workspace. Per-package detail lives in `packages/<name>/CHANGELOG.md` (auto-generated by [Changesets](https://github.com/changesets/changesets)).

The site mirror is at [Changelog](/docs/reference/changelog).

For each release we document:
- Affected packages and their version bumps.
- New public surfaces (with a link to the doc page).
- Known limits introduced by the release (mirrored to [Limits](/docs/reference/limits)).
- Install-blocking issues (e.g. v0.1.1 hot-fix for `workspace:*`).

## Major-bump checklist (what to expect at v1.0)

v1.0 is the production-ready inflection point. Targeted changes:

- **Stability commitment.** v1 surfaces commit to the stricter no-removal-without-RFC contract. Today that's `mochi.launch`, `Session`, `Page`, `Page.humanClick / humanType / humanScroll`, `Session.cookies`, `session.fetch`, `Page.localStorage / sessionStorage`, `Page.grantAllPermissions`, `Page.screenshot`. Anything not on that list at v1.0 freeze is an explicit experimental.
- **Renames (if any) are documented.** Today, no public surface is slated for rename at v1. Internal-only items can churn.
- **Behavioral recorder** â€” `humanClick(sel, { trace })` accepts a recorded trace alongside the synthesizer. The forward-compatible parameter is already in the contract; the recorder lands in v1.x.
- **Profile catalog growth.** Additional real-device captures from the harvester corpus. Placeholder IDs (see below) may flip to real captures.
- **Probe Manifest schema.** Defined at `schemas/probe-manifest.schema.json`. v1.0 freezes the schema version; future bumps follow semver and are tracked in `CHANGELOG.md`.

Anything that would violate an invariant (PLAN.md Â§2 / I-1 through I-8) is by definition not in the v1.0 plan. Migrations across major-bumps will get a dedicated section here when they exist.

## Upgrade-from-v0.1 notes

The v0.2 wave-4 surfaces are **additive**. If you're on `@mochi.js/core` `0.1.0` or `0.1.1` and you upgrade to `0.1.4`, no existing code breaks; you just gain new APIs. Specifically:

- **Audio + canvas precomputed blobs (R-047, R-048)** â€” invisible to your code; the matrix produces them and the inject consumes them. Probes that previously slipped through the audio / canvas surface now match byte-exactly.
- **Cookie persistence** (`Session.cookies.{save,load}`) â€” additive.
- **DOM storage** (`page.localStorage.{get,set}`, `page.sessionStorage.{get,set}`) â€” additive; previously you'd `page.evaluate(() => localStorage.foo)`.
- **`page.grantAllPermissions(opts?)`** â€” additive.
- **`page.screenshot(opts?)`** â€” was `NotImplementedError` in earlier 0.1.x; now returns `Uint8Array` (default) or `string` (`encoding: "base64"`). If you had `try { ... } catch (NotImplementedError)` blocks around screenshots, the catch block is now dead code.
- **Init-script dual mechanism** â€” invisible to your code; closes the `addScriptToEvaluateOnNewDocument` source-attribution leak via `Fetch.fulfillRequest` body splice on Document responses.
- **Turnstile auto-click** â€” opt-in via `challenges: { turnstile: { autoClick: true } }`. Default `false`. No-op if you don't pass it.

See the [Changelog](/docs/reference/changelog) for the per-version detail.

## Workspace-protocol gotcha (pre-v0.1.1)

If you installed `@mochi.js/core@0.1.0` and saw `Workspace dependency not found` errors, that's the v0.1.0 publish bug: `workspace:*` references made it into the published `package.json`. Changesets / `npm publish` does not rewrite `workspace:*` to concrete semver â€” that is a pnpm/yarn courtesy npm doesn't replicate.

The fix is in v0.1.1+ via `scripts/rewrite-workspace-deps.ts` (publish-time pre-hook). **Upgrade to â‰Ą v0.1.1**:

```sh
bun add @mochi.js/core@latest @mochi.js/cli@latest
```

Affected packages bumped to `0.1.1`: `@mochi.js/core`, `@mochi.js/cli`, `@mochi.js/inject`, `@mochi.js/net`, `@mochi.js/behavioral`, `@mochi.js/harness`, `@mochi.js/profiles`. `@mochi.js/consistency` and `@mochi.js/net-rs` were leaf packages with no internal deps; they remain at `0.1.0`.

## Profile-ID stability

The profile catalog is **split into two buckets** with different stability guarantees.

**Real-device baselines (stable IDs â€” depend on these):**
- `mac-m4-chrome-stable`
- `mac-chrome-stable`
- `mac-chrome-beta`
- `windows-chrome-stable`
- `linux-chrome-stable`
- `mac-brave-stable`

These are captured by `mochi capture` on real hardware, filtered by FingerprintJS Pro `suspectScore <= 20`, validated by harness Zero-Diff round-trip. The IDs are stable. Future patch / minor releases will not rename them; new captures of the same device class extend (e.g. a Mac M5 lands at `mac-m5-chrome-stable`, not by replacing `mac-m4-chrome-stable`).

**Placeholders (do NOT depend on these IDs â€” they may flip to real captures or be renamed):**
- `mac-m2-chrome-stable`
- `mac-intel-chrome-stable`
- `win11-edge-stable`

These resolve to a generic synthesis that's consistency-clean but doesn't match any specific captured device. When a real Mac M2 / Mac Intel / Edge capture lands in the harvester corpus, the placeholder ID may either (a) become a real capture under the same ID, or (b) be renamed (e.g. `mac-m2-air-chrome-stable` if the captured machine is specifically a Mac M2 Air, leaving `mac-m2-chrome-stable` as the new placeholder for the device class).

If you want stability today, **use the six real-device IDs**. Track new captures via the [Limits](/docs/reference/limits) page.

## Schema versioning

The `MatrixV1` and `ProbeManifestV1` schemas carry their version in the type name. A future `MatrixV2` would land alongside `MatrixV1` (not as a replacement); old captures continue to deserialize cleanly. Same shape for `ProfileV1` â†’ `ProfileV2`.

`@mochi.js/profiles` ships per-profile JSON validated against `schemas/profile.schema.json` and `schemas/probe-manifest.schema.json`. Schema diffs are documented in PLAN.md Â§6.

## What to do when a release breaks Zero-Diff

If a CI run on your fork breaks Zero-Diff after upgrading mochi, the canonical move is:

1. Check `packages/profiles/data/<id>/expected-divergences.json` for the affected profile â€” the diff may be a new GUID-class field that needs allowlisting.
2. Re-capture the baseline if you own the device â€” `mochi capture --output packages/profiles/data/<id>/`.
3. Open an issue tagged `harness`. The harness is the gate per invariant I-7; a Zero-Diff regression is treated as a real bug.

See [Probe Manifest](/docs/concepts/probe-manifest) for the harness-gate flow.

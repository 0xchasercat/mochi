A profile in mochi is a captured snapshot of a real device — `mac-m4-chrome-stable` is what an actual MacBook running stock Chrome reports across every fingerprint surface. Profiles aren't synthesized. They're captured by a known person on a known machine on a known date and committed to the repo with a `PROVENANCE.md`. The [Consistency Engine](/docs/concepts/consistency-engine) is what turns a profile + a seed into a relationally-locked Matrix; profiles are the input data layer.

PLAN.md §12.

## The v1 catalog

Six real-device baselines ship today, each captured against real Chrome on real hardware, filtered by FingerprintJS Pro `suspectScore <= 20`, and validated by the harness round-trip:

| Profile id | Device | Browser |
|---|---|---|
| `mac-m4-chrome-stable` | MacBook (Apple Silicon, M4) | Chrome stable |
| `mac-chrome-stable` | macOS | Chrome stable |
| `mac-chrome-beta` | macOS | Chrome beta |
| `windows-chrome-stable` | Windows 11 | Chrome stable |
| `linux-chrome-stable` | Linux x86_64 | Chrome stable |
| `mac-brave-stable` | macOS | Brave stable |

Other ids in the catalog (`mac-m2-chrome-stable`, `mac-m1-chrome-stable`, `mac-intel-chrome-stable`, `win11-chrome-stable`, `win11-edge-stable`) still resolve to a generic Linux placeholder. The Matrix is still relationally locked, but the surface values aren't from a real capture — `gpu.renderer` is a generic "Intel Iris Xe Graphics" string, fonts default to a Linux baseline list. We say so on the [Limits page](/docs/reference/limits) rather than pretending to ship them. The placeholder lets you exercise the API surface end-to-end before the captures land.

The full id list is exported as `KNOWN_PROFILE_IDS` from `@mochi.js/profiles`:

```ts
import { KNOWN_PROFILE_IDS, type ProfileId } from "@mochi.js/profiles";

KNOWN_PROFILE_IDS;
// readonly [
//   "mac-m4-chrome-stable", "mac-m2-chrome-stable", "mac-m1-chrome-stable",
//   "mac-intel-chrome-stable", "win11-chrome-stable", "win11-edge-stable",
//   "linux-chrome-stable", "mac-chrome-stable", "mac-chrome-beta",
//   "windows-chrome-stable", "mac-brave-stable",
// ]
```

## The `ProfileV1` schema

A `ProfileV1` declares the *capabilities* of a device class. The schema lives at [`schemas/profile.schema.json`](https://github.com/0xchasercat/mochi/blob/main/schemas/profile.schema.json); the TypeScript type is generated by `bun run codegen` and re-exported from both `@mochi.js/consistency` and `@mochi.js/profiles`. Selected fields:

```ts
interface ProfileV1 {
  id: string;                              // "mac-m4-chrome-stable"
  version: string;                         // semver — bumps on re-capture
  engine: "chromium";
  browser: { name: "chrome" | "edge" | "brave"; channel: "stable" | "beta"; minVersion: string; maxVersion: string };
  os: { name: "macos" | "windows" | "linux"; version: string; arch: "arm64" | "x64" };
  device: { vendor: string; model: string; cpuFamily: string; cores: number; memoryGB: number };
  display: { width: number; height: number; dpr: number; colorDepth: number; pixelDepth: number };
  gpu: { vendor: string; renderer: string; webglUnmaskedVendor: string; webglUnmaskedRenderer: string; webglMaxTextureSize: number; webglMaxColorAttachments: number; webglExtensions: string[] };
  audio: { contextSampleRate: number; audioWorkletLatency: number; destinationMaxChannelCount: number };
  fonts: { family: string; list: string[] };
  timezone: string;                        // IANA, "America/Los_Angeles"
  locale: string;                          // BCP-47, "en-US"
  languages: string[];
  behavior: { hand: "left" | "right"; tremor: number; wpm: number; scrollStyle: "smooth" | "stepped" | "inertial" };
  wreqPreset: string;                      // DEPRECATED in 0.7 — runtime ignores; kept in the schema for back-compat.
  userAgent: string;
  uaCh: Record<string, string>;
  entropyBudget: { fixed: string[]; perSeed: string[] };
}
```

`entropyBudget.fixed` lists the surfaces that are constants for this profile across all seeds (`gpu.*`, `audio.contextSampleRate`, `fonts.list`, `display.dpr`). `entropyBudget.perSeed` lists the surfaces that are deterministic-but-vary-per-seed within bounds the profile declares (`display.width`, `display.height`, `timezone`, `languages[0]`). A seed picks one combination from the per-seed entropy space.

`browser.minVersion` / `maxVersion` declare which Chromium-for-Testing versions the profile is verified against. `mochi.launch` checks the binary version against this range and emits a warning if outside (errors only with `strict: true`).

## On-disk layout

Each profile lives at `packages/profiles/data/<id>/`:

```
packages/profiles/data/mac-m4-chrome-stable/
├── profile.json                  ProfileV1 — the capability declaration
├── baseline.manifest.json        ProbeManifestV1 — captured from the real device
├── audio/
│   ├── 44100.bin                 OfflineAudioContext bytes per sample-rate (R-047)
│   └── 48000.bin
├── canvas/
│   ├── default.json              Per-probe canvas synthesis triple (R-048)
│   └── creepjs.json
├── expected-divergences.json     Known JS-only-uncoverable divergences (intentional)
└── PROVENANCE.md                 Capturer + machine + date
```

`baseline.manifest.json` is the [Probe Manifest](/docs/concepts/probe-manifest) the harness diffs against. `audio/*.bin` and `canvas/*.json` are the byte-accurate captures consumed by R-047 and R-048 — surfaces that can't be derived from primitives.

## The `mochi capture` workflow

Capture must run against an *unmodified* browser binary on a real machine. The CLI flow:

```sh
mochi capture \
  --profile-id mac-m4-chrome-stable \
  --browser /Applications/Google\ Chrome.app/Contents/MacOS/Google\ Chrome \
  --out packages/profiles/data/mac-m4-chrome-stable
```

What runs:

1. Spawn the user-supplied browser with a clean user-data-dir. Capture flows pass `bypassInject: true` to `mochi.launch` so neither `buildPayload` nor the `Fetch.fulfillRequest` body splice run — the browser reports its bare, native fingerprint.
2. Drive to the local probe-page fixture (`tests/fixtures/probe-page.html`).
3. Capture the Probe Manifest using a CDP attach.
4. Extract device-class facts (OS, browser version, GPU strings, audio sample rate, fonts list, etc.) into `profile.json`.
5. Capture audio fingerprint bytes for each known sample-rate of interest into `audio/*.bin`.
6. Capture canvas fingerprint maps for each known test payload into `canvas/*.json`.
7. Write `PROVENANCE.md` from interactive prompts (capturer, machine model, date).

Re-capturing — when Chrome ships a new minor and the canvas hash drifts — bumps the profile version (`mac-m4-chrome-stable v1.0.0 → v1.1.0`). Old versions remain in the catalog with `deprecated: true` for one release window, then drop.

## What "real-device" means

Three checks turn a raw capture into a committed profile:

1. **Provenance verified.** `PROVENANCE.md` names the capturer, the machine model, the date. A maintainer reviews at PR time; CI cannot verify provenance.
2. **FingerprintJS Pro `suspectScore <= 20`.** The capture session is run through FingerprintJS Pro; the suspectScore (a residual-trust metric) must be below 20. A residential IP, a real device, and a session with browsing history pre-warmed clear this.
3. **Harness round-trip.** The capture produces a `baseline.manifest.json`. A subsequent `mochi.launch({ profile: <id>, seed: "harness-canary" })` followed by `capture(session, { fixtureUrl: ... })` produces a manifest that differs from the baseline by zero material entries (only `guid-class` and `intentional` allowed).

Profiles that fail any of these don't ship. The catalog is small on purpose. PLAN.md I-8 — honesty over marketing.

## `getProfile` and the resolution path

```ts
import { getProfile } from "@mochi.js/profiles";

const profile = await getProfile("mac-m4-chrome-stable");
// → fully-resolved ProfileV1 (loads from packages/profiles/data/<id>/profile.json)
```

`getProfile` lands in phase 0.4 alongside the first captured baseline. Until then, `mochi.launch({ profile: <string> })` resolves a string id directly: it goes through a placeholder resolver that synthesizes a generic Linux ProfileV1 with the requested id stamped on it. That placeholder feeds the consistency DAG identically — the *Matrix* is real and relationally locked — but the *surface values* are generic. Useful for API exploration, not for stealth.

You can also pass an inline `ProfileV1` object to `mochi.launch`:

```ts
import { mochi } from "@mochi.js/core";
import type { ProfileV1 } from "@mochi.js/consistency";

const inlineProfile: ProfileV1 = { /* hand-built or loaded from your own capture */ };
const session = await mochi.launch({ profile: inlineProfile, seed: "user-12345" });
```

If you hand-build a profile, you own the consistency contract. The DAG still runs and produces a Matrix; the *Matrix* is internally consistent. But if your inline `userAgent` says Mac and your `os.name` says `linux`, the resulting Matrix is internally inconsistent and the harness will refuse to certify it (PLAN.md I-5).

## Per-profile `expected-divergences.json`

Each profile carries a list of known JS-only-uncoverable divergences:

```jsonc
// packages/profiles/data/mac-m4-chrome-stable/expected-divergences.json
[
  "webrtc.localIp",
  "navigator.connection.rtt"
]
```

Anything in this list is categorized as `intentional` by the [harness diff](/docs/concepts/probe-manifest). Adding to it requires a corresponding entry in [`docs/limits.md`](/docs/reference/limits) — the document is written so that pretending we don't know about a gap is harder than admitting it.

## What we don't ship as profiles

- **Mobile / touch profiles** — v1 is desktop Chromium-family only. Touch gestures, sensor APIs, and OS-level touch-event sequencing land in v2.
- **Safari / Firefox profiles** — cross-engine FPU / JIT divergence is out of v1 scope. v2+ research.
- **Custom-OS Chromium variants** — ChromeOS, certain Linux distros with distinctive font sets. Captures welcome via PR; provenance discipline applies.
